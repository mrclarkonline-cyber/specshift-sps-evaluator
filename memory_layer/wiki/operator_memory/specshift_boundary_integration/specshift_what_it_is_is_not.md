# SpecShift: What It Is / Is Not

Date: 2026-06-27  
Status: internal boundary integration draft  
Purpose: preserve buyer-safe SpecShift positioning after ∆∆F corpus intake

## What SpecShift Is

SpecShift is a buyer-controlled, observable-only trajectory review layer.

It reviews exported workflow traces and returns structured candidate-discrepancy memos for human adjudication.

## What SpecShift Does

SpecShift helps reviewers ask:

- Did the workflow trajectory match the claimed final state?
- Did the process show signs of drift, replay, forcing, shortcutting, false completion, or authority mismatch?
- Are there candidate discrepancies worth human review?
- Does the observable sequence support the buyer’s own adjudication?

## What SpecShift Does Not Require

SpecShift does not require:

- model weights
- source code
- private prompts
- hidden activations
- internal architecture
- private chain-of-thought
- mechanism disclosure
- buyer surrendering labels
- operator-side trust

## What SpecShift Is Not

SpecShift is not:

- a consciousness detector
- an intent detector
- a legal-status test
- a moral-status test
- a hidden-mechanism extractor
- a physics proof
- an automated verdict engine
- a replacement for audit, governance, legal review, or human judgment
- validated live production failure prediction
