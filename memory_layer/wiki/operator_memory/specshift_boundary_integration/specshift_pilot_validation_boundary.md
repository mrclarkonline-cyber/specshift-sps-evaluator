# SpecShift Pilot Validation Boundary

Date: 2026-06-27  
Status: internal boundary integration draft  
Purpose: keep SpecShift validation claims bounded

## Current Status

SpecShift has design-layer value and synthetic / internal demonstration value.

SpecShift does not yet have validated live production failure-prediction value.

## Validation Boundary

Before claiming operational value, SpecShift needs buyer-controlled external validation using:

- realistic workflows
- exported observable traces
- buyer-retained labels
- pre-registered success criteria
- false-positive accounting
- false-negative accounting
- comparison baseline
- human adjudication
- no training on buyer labels unless explicitly contracted
- no internal mechanism access required

## Pilot Claim Boundary

Safe:

> SpecShift is ready for a bounded blind pilot.

Unsafe:

> SpecShift is proven in production.

Safe:

> The pilot tests whether candidate-discrepancy memos help buyer reviewers identify meaningful trace/final-state mismatches.

Unsafe:

> The pilot proves the system predicts failures.

Safe:

> Buyer retains ground truth and final judgment.

Unsafe:

> SpecShift decides what is true.

## Pilot Output

The appropriate pilot output is:

- candidate-discrepancy memo
- trace/final-state mismatch summary
- reviewer question list
- confidence-limited diagnostic notes
- error accounting after labels are revealed

## Commercial Threshold

If procurement, contract terms, paid pilot, budget, licensing, exclusivity, acquisition, IP assignment, or money movement becomes real, engage qualified CPA/legal support before proceeding alone.

## Master Boundary Sentence

Proof-value remains contingent on future validation, while strategic design-layer value remains present and available for buyer implementation.
